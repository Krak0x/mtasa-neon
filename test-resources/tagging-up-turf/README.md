# Tagging Up Turf co-op prototype

This resource is the first vertical slice for Neon's planned story runtime. It recreates the gameplay route of GTA: San Andreas' `SWEET1` mission as an authoritative MTA Lua prototype. Its positions and stage order follow the decompiled `main.scm` mission, while its current actor, camera, and tag behaviors are deliberately replaceable experiments.

The target architecture is documented in [`STORY_RUNTIME.md`](../../STORY_RUNTIME.md). The long-term implementation will keep mission/SCM orchestration server-authoritative while moving reusable GTA tasks, tags, camera behavior, and other engine semantics into generic Neon C++ APIs.

## Play

1. Start the resource: `start tagging-up-turf`.
2. Connect one to three players.
3. Run `/tagup`. The player who starts the mission is the driver/leader.
4. Follow GTA's native translated mission prompts and map markers. Hold fire with the spray can while aiming at each marked tag.

Development commands:

- `/tagupsweet1a` is an explicit alias for the full start. It loads and plays GTA's native `SWEET1A` DAT/CUT/IFP scene, then continues directly into the world intro and mission.
- `/tagupidlewood` starts inside the Greenwood at the real Idlewood arrival gate. The car begins slightly rolled above the road so logs can prove that the scene remains blocked after the first wheel contact and starts only when exact opcode `09D0` reports all four contacts.
- `/tagupdeparture` starts inside the Greenwood at the real Ballas arrival gate, with only the earlier tags completed. It uses the same rolled all-wheels test and preserves the actual camera, leave-car, `SWE1_AV`, natural audio-finish, and `05D2` flow.
- `/tagupballas` starts immediately outside the SCM gang-camera gate with the earlier tags completed, while preserving the real Ballas spawn, chat, camera, approach, follow, timer, and attack flow.
- `/taguppickup` starts directly at the post-rooftop recording scene. It holds the party safely above the destination until each client proves the local collision is streamed, then preserves the `0A0B` directional load, fixed camera, two `09F7` horns, `SWE1_BH`, and cleanup flow.
- `/tagupfinal` starts directly at the Grove Street finale, stages CJ and Sweet from the real arrival position, and preserves the complete camera, dialogue, animation, walk-away, skip, and cleanup flow.
- `/tagupskip` advances the current stage (leader only).
- `/tagupabort` aborts and restores every participant's previous state.
- `/taguptrace [on|off]` or `F7` controls the local instruction-trace overlay.
- `/taguptracepreview` loads a presentation-only sample sequence for visual QA.

## Instruction trace component

`instruction_feed.lua` provides a generic client-side `TAGUP_TRACE` namespace. It is loaded before `client.lua` and receives presentation-only updates from the mission: toggling it cannot affect stage progression. The overlay is invisible by default, keeps completed instructions dimmed, highlights the current instruction and interpolated progress, and shows future instructions as queued.

The live sequence labels each entry honestly as a verified native opcode/task, SCM flow or condition, co-op adaptation, or temporary Lua substitute. Fast synchronous instructions can move directly into the dimmed history without being held on screen; the trace never delays gameplay just to animate the UI. `/taguptracepreview` remains a purely local visual sample.

The panel reuses the running `chatbox` resource's `fonts/Arial.ttf` through MTA's cross-resource path support. It creates larger dedicated header, instruction, detail, status, and footer fonts and falls back to built-in fonts if `chatbox` is unavailable. Its black, warm-orange, and ivory treatment intentionally follows San Andreas' original mission UI rather than Neon's modern green diagnostic styling.

```lua
TAGUP_TRACE.setSequence(steps, {title = "TAGGING UP TURF", subtitle = "CO-OP TRACE"})
TAGUP_TRACE.setCurrent(stepId [, detail])
TAGUP_TRACE.skipTo(stepId [, detail])
TAGUP_TRACE.setStatus(stepId, "queued" | "active" | "done" | "failed" | "skipped" [, detail])
TAGUP_TRACE.fail(stepId [, detail])
TAGUP_TRACE.setProgress(stepId, progress [, detail]) -- progress accepts 0..1 or 0..100
TAGUP_TRACE.reset()
TAGUP_TRACE.toggle([visible])
```

Sequence entries may be strings or tables containing `id`, `title`, `detail`, `status`, and `progress`. The component only draws local DX primitives and owns no timers, elements, mission tokens, or server events. The detailed native-task feed is intentionally available on the mission leader/syncer, which is the client that actually executes and observes Sweet's GTA tasks; other party members do not fabricate those observations locally.

## Current scope

The prototype includes GTA's native `SWEET1A` file cutscene, the following world intro camera, shared co-op objectives, Sweet and his Greenwood, five sprayable tags, a Sweet demonstration, synchronized Ballas combat, mission failure, the return drive, rewards, and restoration of each player's pre-mission state. It uses dimension 4101 so it can run beside other test resources. Its former always-on DX title/objective banner, tag percentages, spray hints, skip hints, and transient French status messages have been removed. Runtime instructions, help, dialogue subtitles, and mission passed/failed banners now use GTA's native GXT block and HUD queues with the original `SWEET1` keys and timings. The F7 instruction trace remains a separate, disabled-by-default diagnostic overlay.

The full start now mirrors the installed `main.scm` boundary before synchronized mission entities exist. Every participant acquires a resource-owned native file-cutscene lease, loads `SWEET1A`, starts only after the server readiness barrier, and reports native camera-spline completion. GTA's own local skip consumption is suppressed for this managed playback; only the leader's original native skip input is sent to the server and broadcast to the party. Every client performs the SCM's immediate black fade and deletes native cutscene data before the server creates Sweet, the Greenwood, and the tags. Abort, failure, disconnect, resource stop, camera takeover, timeout, and stage replacement all use the same native cleanup path. Manual single-player validation completed the full native playback at the expected speed, showed the animated spray prop, released the cutscene lease, and continued through `SWE1_AA` to `SWE1_AE` with camera and audio cleanup acknowledged. Multi-participant validation remains pending.

Navigation now follows the SCM's blip lifecycle: Sweet's Greenwood has an attached friendly blue blip whenever the player must return to it, driving stages replace it with the default cream destination blip, and every active tag has its exact green coordinate blip until that tag reaches 100 percent. The rooftop tag stays hidden until the player reaches the original approach point.

Tag hits now come from Neon's resource-owned native gang-tag service. GTA's real spray-can shot path selects the object, advances its logical alpha by the original increment of 8, updates the native Grove material, returns the original completion state at 255, and emits `onClientObjectGangTagProgress`. The server validates the reporting player or Sweet syncer, weapon, mission stage, object, distance, and exact native delta, then mirrors one authoritative alpha step to the party. It deliberately does not impose a wall-clock delay between valid reports because `CShotInfo::Update` can process several spray shots in one frame. Lua no longer polls fire input, guesses a nearby tag, or advances Sweet's demonstration with a timer. Stream-out, object recreation, resource stop, and conflicting ownership are handled by the generic C++ service while MTA's unsafe global `CTagManager` remains disabled.

The SCM bytecode itself is not executed. Lua keeps the server-authoritative control flow, while verified native services now provide the reusable GTA camera, mission-text, mission-audio, task, animation, and recorded-car behavior used by the implemented slices. This resource is a regression harness and architecture probe, not the intended mission-by-mission implementation strategy.

Sweet's first Idlewood demonstration now exercises Neon's native `setPedExitVehicle`, `setPedGoTo`, `setPedShootAt`, and `setPedEnterVehicle` paths using the `SWEET1` profiles. Arrival uses the SCM's 4-by-4-by-4 locate box and exact client-syncer `09D0` all-four-contact predicate rather than the former 11 m sphere and broad MTA ground approximation, and its coordinate blip uses GTA's default cream destination colour rather than the friendly blue reserved for Sweet's car. GTA normally queues `SWEET1_CA` then `SWEET1_AR` immediately before fixed-camera staging; the co-op harness preserves that order but reserves the same two native handles as soon as the party enters the Greenwood. This compatibility prefetch lets unusually slow yet structurally valid replacement banks perform their cold disk load during the drive, then transfers the handles unchanged into the arrival scene. While the resource polls both handles at the barrier, Neon periodically re-arms a native hardware request that GTA accepted into its mission-audio link but silently failed to queue. Silent tracks still follow the normal loaded/play/finished lifecycle, and the server starts only after every client reports both native handles loaded. Release also resets GTA's retained event fingerprint after cancelling the sample, so repeated server reconnects cannot strand all four native slots as apparently foreign ownership. On the driver/syncer, that generic lease sets GTA's verified `bPlayerSafe` pad bit, making the Greenwood receive zero throttle, full brake, handbrake, and the native `0.28` speed clamp instead of a mission-specific velocity reset or freeze. The scene reproduces the three fixed shots, black-screen actor staging, native fades, widescreen, control inhibition, and simultaneous 10-second position/target tracks from the SCM. Sweet receives his native leave-car task first; 600 ms later each player requests the same native task locally, while the authoritative position fallback runs only after the fade is fully black. Only the leader may request a skip, but the server applies it to the whole party and restores gameplay under the black fade before every client fades back in.

The vehicle exit is no longer a server-side removal: the leader/syncer requests MTA's authoritative exit lifecycle, observes `TASK_COMPLEX_LEAVE_CAR`, and the server independently confirms `onVehicleExit` before it advances. Sweet then walks to `2100.48, -1649.14, 12.47`, applies shooting rate `100` and accuracy `90`, and starts the verified native shoot task. His mission-actor policy suspends MTA's ordinary player-weapon target substitution, which otherwise replaces the task's tag coordinate because MTA represents script peds as `CPlayerPed` instances. Reaching 100% interrupts that task and waits the SCM's following 1000 ms. The server then plays the existing synchronized `GRAFFITI_CHKOUT` animation with parameters equivalent to blend delta `4.0`, non-looped, interruptible, no root motion, no frozen final frame, and waits for the syncer to observe its natural end. Sweet walks back, every client plays `SWE1_CA` to natural completion, and all camera/audio releases are acknowledged before control returns. Sweet enters MTA passenger seat `1` (SCM passenger index `0`) concurrently with the two playable Idlewood tags. Facial lip-sync remains pending because Neon does not yet expose `CTaskComplexFacial::SetRequest`; the dialogue audio and timing no longer use substitute timers.

At the Ballas destination, the old `removePedFromVehicle` plus Sweet position teleport has also been removed. The trigger now uses the SCM's 4-by-4-by-4 locate box and exact client-syncer `09D0` all-four-contact gate instead of the former 13 m sphere and broad MTA ground approximation. Every participant then requests event `37420`, the original `SWE1_AV` line, and acquires an independent resource-owned native camera lease. The server waits for all cameras before starting the synchronized native leave-car lifecycle. Only after each client has requested that task does it poll the audio handle, matching the SCM order. A second readiness barrier synchronizes playback after every native handle is loaded. The scene therefore reproduces the fixed position, point-at target, widescreen state, control inhibition, minimum 100 ms lead-in, native audio playback, and natural-finish wait without racing another resource's camera, controls, or audio slots. Sweet carries a synchronized mission-actor policy which every client applies through `setPedMissionActor`; this reproduces SCM `CREATE_CHAR` and prevents GTA's four-second ambient passenger-abandonment sequence. Only after the server confirms that all players are outside and every client has observed `SWE1_AV` finish does it persistently assign the leader as syncer for both Sweet and the Greenwood. The leader then calls `setPedDriveWander(sweet, greenwood, 20.0, "avoid_cars")`, observes `TASK_COMPLEX_CAR_DRIVE_WANDER`, and completes the SCM's following 1000 ms window before every client releases its camera and audio leases and the Ballas tag stage begins. The verified installed `main.scm` contains no active seat-shuffle opcode here: Sweet intentionally remains passenger while GTA's autopilot drives the empty-driver vehicle.

The two-Flats encounter follows the separate one-shot gates in `SWEET1`: the server creates unarmed models 102 and 103 at their exact SCM positions when the leader enters the outer 50-by-50 box, then queues their paired native `CTaskComplexPartnerChat` tasks in SCM order through GTA's real `CEventScriptCommand` path. GTA's `CREATE_CHAR` opcode adds `1.0` to the script Z before placing an actor, while MTA's `createPed` consumes its Z directly, so the resource applies that conversion without changing the source SCM coordinates. This test installation retains the ped-speech containers but strips their samples. GTA consequently reserves an audio conversation successfully and then ends every `CTaskComplexChat` immediately because `GetPedTalking()` remains false. The resource passes `conversationEnabled=false`, which reproduces the native post-reservation-failure state rather than constructing a different task state: GTA first normalizes the direction counter to `4`, then Neon clears only the conversation flag before dispatch. PartnerChat therefore selects its own timed `SimpleChat`/`StandStill` fallback while retaining GTA's native actor positioning and idle state machine. The camera gate remains closed until the syncer observes both PartnerChat tasks alive for ten consecutive samples; accepting the two API calls alone is not treated as ready. The shot then starts only inside the 20-by-17 box while both actors are alive. All participants acquire the native camera lease before the server begins the exact fixed shot at `2400.3840, -1472.2081, 23.9349`, followed by the non-skippable 500 ms lead-in and the 6500 ms skip window. Only the leader may request a skip, but the server applies that decision to the whole party. Natural completion and skip share two barriers: every client first proves its lease is still valid, then releases it and acknowledges the actual restoration. `/tagupballas` also keeps each teleported player frozen until that player's client confirms the destination ground collision and the server places the ped one metre above the measured surface.

After the shot, the leader entering the original 5-by-5 box gives each living Flat the verified scripted-speech mute, zero-duration StandStill, and repeating `TASK_GOTO_CHAR_OFFSET` sequence at radius `0.5` and angles `90` or `270`. The local locate report is retried until the authoritative server position enters the same box, and the resulting follow event serves as its acknowledgement. Neon uses GTA's global mission sequence pool and `CTaskComplexUseSequence`, so the offset is recalculated by the native task rather than by Lua polling. The alley tag reaching 100 percent or the preserved SCM timer passing 5000 ms replaces those sequences with native `CTaskComplexKillPedOnFoot` tasks targeting the mission leader. The rooftop transition waits for the syncer to acknowledge that native attack assignment, preserving the SCM ordering across the network. `SWE1_BA` and `SWE1_BE` remain optional exactly like the SCM: each plays only if its native mission-audio handle is loaded, and their silent mission-audio tracks never block movement, combat, or mission progress. That optional mission-audio behavior is separate from the ambient ped conversation used by PartnerChat, whose empty samples do change native task duration as documented above. The old 180 ms Lua aiming, firing, rotation, and forward-control substitute has been removed.
